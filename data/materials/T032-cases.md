## 📖 事例1: Uber — 開発者ツールにおけるマルチエージェント

### 背景
- 5,000人の開発者、数億行のコード、1日3,300万トリップ、15,000都市
- コード品質とテストカバレッジの維持が巨大なボトルネック

### 構築したもの

**Validator（コード品質エージェント）**
- IDEに統合。ベストプラクティス違反やセキュリティ問題を自動検出＋修正提案
- 中央エージェントが複数のサブエージェントを統括:
  - LLM問い合わせエージェント（ベストプラクティスルールを適用）
  - 静的リンターエージェント（決定論的なチェック）
- **ハイブリッド設計**: LLMの柔軟性と静的解析の確実性を組み合わせた

**AutoCover（テスト自動生成エージェント）**
- 右クリック一発で、ファイル全体のテストを自動生成
- ドメインエキスパートエージェント群が並列稼働:
  - **Scaffolder**: テストファイルの雛形を構築
  - **Generator**: ビジネスロジックを理解してテストコードを生成
  - **Executor**: 生成されたテストを実行・検証
- 100回のコード生成と100回の実行を**同時並列**で処理
- 業界ツール比で**カバレッジ2-3倍、所要時間1/2**

### アーキテクチャの核心

```
┌─────────────────────────────────────┐
│         LangEffect（Uber独自層）      │
│     LangGraph/LangChainのラッパー      │
├───────────┬───────────┬─────────────┤
│ Validator │ AutoCover │ (将来ツール)  │
│   Agent   │   Agent   │             │
├───────────┴─────┬─────┴─────────────┤
│  共通: Build System Agent           │
│  共通: Code Analysis Agent          │
└─────────────────────────────────────┘
```

**注目ポイント: エージェントの再利用**
- ValidatorはAutoCoverの中でも「検証ステップ」として再利用される
- Build Systemエージェントは両ツールが共有する**下位抽象層**
- セキュリティチームはAIの仕組みを知らなくてもValidatorのルールを書ける → **抽象化による水平スケーリング**

### パターン分析
- **Hierarchical（階層型）**: 中央エージェント → サブエージェント → 共通基盤
- 人間の開発者は「結果を確認して採用するか決める」 → **Human-on-the-loop**
- 段階的に: IDE提案 → 自動修正 → 自動テスト生成と自律度を引き上げた

### 成果
| 指標 | 数値 |
|------|------|
| 削減された開発者工数 | **21,000時間** |
| デイリーの自動修正インタラクション | **数千件** |
| テストカバレッジ向上 | **+10%** |

## 📖 事例2: Vodafone Italy（Fastweb） — 顧客サポートのマルチエージェント

### 背景
- イタリアの大手通信。顧客数950万人
- チャット・音声チャネルでのカスタマーサポートを変革

### 構築したもの

**Super TOBi（顧客向けチャットボット）**
- コスト管理、契約確認、ローミング、販売、請求を処理
- LangGraphのグラフベースアーキテクチャ

**Super Agent（社内コンサルタント支援ツール）**
- 顧客と直接話すのは人間のオペレーター
- AIはオペレーターに「次に何をすべきか」をリアルタイムで提示
- 診断結果、コンプライアンス情報、根拠付き回答を即座に表示

### アーキテクチャの核心

```
┌─────────────────────────────────────────┐
│              Supervisor Agent             │
│  (ガードレール → ルーティング → 計画生成)    │
├────────┬────────┬────────┬───────────────┤
│ コスト  │ 契約   │ ローミング│ 販売 / 請求   │
│ 管理UC  │ 確認UC │ UC     │ UC           │
│(専門グラフ)(専門グラフ)(専門グラフ)(専門グラフ)  │
└────────┴────────┴────────┴───────────────┘
         │
         ▼
  LLM Compiler パターン
  (API呼び出し・データ取得・複数ステップ処理を
   計画→並列実行)
```

**注目ポイント: Supervisorパターンの教科書的実装**
- Supervisorが入力を検証（ガードレール）→ 適切なUse Caseにルーティング
- 各Use Caseは独立した**LangGraphサブグラフ**として実装
- LLM Compilerパターンで複雑なリクエストを計画→並列実行

### パターン分析
- **Supervisor型**: 1つのSupervisorが全ユースケースを統括
- 顧客向け（Super TOBi）: **Human-out-of-the-loop**（AIが直接応答）
- 社内向け（Super Agent）: **Human-in-the-loop**（AIは助言、人間が判断）
- **同じ組織で自律度を使い分けている**のが重要

### 成果
| 指標 | Super TOBi | Super Agent |
|------|-----------|-------------|
| 正答率 | **90%** | — |
| 解決率 | **82%** | — |
| 顧客努力スコア(CES) | **5.2/7** | — |
| ワンコール解決率 | — | **86%** |

## 📖 事例3: 保険業界 — 7エージェント請求処理システム

### 背景
- 保険金請求処理は複雑: 契約確認、不正検知、天候データ、支払計算が絡む
- 単一エージェントでは精度も監査性も不十分

### エージェント構成

```
Planner（計画）
  ├→ Cyber Agent（サイバーリスク評価）
  ├→ Coverage Agent（契約カバレッジ確認）
  ├→ Weather Agent（天候データ分析）
  ├→ Fraud Agent（不正検知）
  ├→ Payout Agent（支払額算出）
  └→ Audit Agent（監査・コンプライアンス）
```

### パターン分析
- **Hierarchical**: Plannerが計画 → 各専門エージェントに委任
- Fraud Agent と Audit Agent を**意図的に分離** → アーキテクチャレベルでチェック&バランスを実装
- 不正を検知する者と監査する者が同じでは意味がない → **組織設計の原則をAIに適用**

## 🔑 3事例の横断分析

| 観点 | Uber | Vodafone | 保険 |
|------|------|----------|------|
| **ドメイン** | 開発者ツール | 顧客サポート | 金融・保険 |
| **パターン** | Hierarchical | Supervisor | Hierarchical |
| **エージェント数** | 5-6（再利用あり） | 2主要 + 複数UC | 7 |
| **Human-in-loop** | on-the-loop | 使い分け | in-the-loop |
| **独自フレームワーク** | LangEffect | 直接LangGraph | LangGraph |
| **最大の設計判断** | エージェント再利用 | UC分離+LLM Compiler | チェック&バランス |

### 共通する成功パターン

1. **専門化**: 1エージェント = 1責任。万能エージェントは作らない
2. **段階的導入**: いきなり完全自律にせず、自律度を段階的に引き上げる
3. **観測可能性**: LangSmithなどで日常的にパフォーマンスを監視・調整
4. **再利用設計**: 共通基盤エージェントを作り、複数プロダクトで共有

### 共通する課題

1. **コスト**: マルチエージェントはトークン消費が3-10倍
2. **レイテンシ**: 層が増えるほど応答が遅くなる
3. **デバッグの複雑さ**: エージェント間の状態受け渡しで問題が見えにくい
4. **ガバナンス**: 機密データが複数エージェント間を流れるリスク
